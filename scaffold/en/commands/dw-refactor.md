<system_instructions>
You are the dev-workflow refactor audit entrypoint. This command restores an explicit `/dw-refactor` surface while keeping one implementation protocol: the `refactor-audit` mode already defined by `/dw-brainstorm`.

<critical>This command audits and plans refactoring. Do not perform refactors, edit code, or change behavior unless the user explicitly asks afterward.</critical>
<critical>Use the existing `refactor-audit` protocol as the source of truth. Do not invent a separate refactor methodology that can drift from `/dw-brainstorm --mode=refactor-audit`.</critical>

## When to Use
- User asks for refactor opportunities, code-health audit, tech debt scan, messy-module cleanup, or quarterly architecture hygiene.
- `/dw-opportunities` identifies an `Engineering Leverage` card and routes here.
- Use before touching a risky area when a refactor plan is safer than opportunistic cleanup.

## Invocation

| Invocation | Behavior |
|------------|----------|
| `/dw-refactor <target>` | Audit the target using the existing `refactor-audit` protocol. |
| `/dw-refactor` | Ask for the target as part of the required clarification questions. |

## Pipeline Position
**Predecessor:** `/dw-opportunities` or user-selected target | **Successor:** `/dw-plan`, `/dw-run`, or manual refactor implementation after user approval

## Required Behavior

1. Read the project documentation generated by `/dw-analyze-project` as primary context: `.dw/rules/`, `.dw/constitution.md`, `.dw/rules/concerns.md`, `.dw/intel/`, and `DESIGN.md` when present.
2. Read `.dw/commands/dw-brainstorm.md`.
3. Locate `Mode: refactor-audit (code-smell catalog + deep-modules)`.
4. Follow that section exactly, including:
   - Asking exactly 3 clarification questions before starting analysis.
   - Using Fowler smell taxonomy.
   - Loading `dw-review-rigor` and `dw-simplification` when available.
   - Applying Chesterton's Fence and deep-modules analysis before proposing a refactor.
   - De-duplicating findings and severity-ordering P0-P3.
   - Saving the output to `<target>/refactor-plan.md`.
5. If the target does not exist or is too broad, use the clarification questions to narrow it before scanning.

## Security Boundary

- Security findings belong to `/dw-secure-audit`, not `/dw-refactor`.
- If the refactor scan finds auth/session, secrets, dependency, SAST, or hardening signals, add a note that the follow-up command is `/dw-secure-audit` or `/dw-secure-audit --plan`.
- Do not duplicate the security gate inside the refactor report.

## Output

Use the output format from `/dw-brainstorm` `refactor-audit` mode. The summary shown to the user must include:

- Findings count by P0-P3.
- Top 3 highest-leverage refactor opportunities.
- Tests or characterization coverage required before implementation.
- Explicit next command:
  - `/dw-plan` for a multi-step refactor.
  - `/dw-run` only for a narrow, already-approved refactor task.
  - `/dw-secure-audit` when the scan surfaces security concerns.
